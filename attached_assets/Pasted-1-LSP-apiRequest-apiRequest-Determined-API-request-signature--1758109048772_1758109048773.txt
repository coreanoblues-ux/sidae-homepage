1) LSP 타입 오류(apiRequest) 근본 해결

증상: “apiRequest 함수의 타입이 맞지 않음”, “Determined API request signature” 류 메시지

src/lib/api.ts (새 파일로 교체 생성)

// src/lib/api.ts
export type ApiMethod = 'GET' | 'POST' | 'PUT' | 'PATCH' | 'DELETE';

type Primitive = string | number | boolean | null | undefined;

export type Query = Record<string, Primitive | Primitive[]>;

export interface ApiRequestInit<TBody = unknown> extends Omit<RequestInit, 'body' | 'method'> {
  method?: ApiMethod;
  body?: TBody; // JSON | FormData | undefined
}

function toQueryString(q?: Query) {
  if (!q) return '';
  const usp = new URLSearchParams();
  Object.entries(q).forEach(([k, v]) => {
    if (Array.isArray(v)) v.forEach(x => usp.append(k, String(x)));
    else if (v !== undefined && v !== null) usp.set(k, String(v));
  });
  const s = usp.toString();
  return s ? `?${s}` : '';
}

function isFormData(x: unknown): x is FormData {
  return typeof FormData !== 'undefined' && x instanceof FormData;
}

/**
 * 안전한 API 요청 래퍼 (완전 제네릭)
 * 사용 예)
 *   const user = await apiRequest<User>('/api/me');
 *   const created = await apiRequest<Post, NewPost>('/api/posts', { method:'POST', body: newPost });
 *   const list = await apiGet<Post[]>('/api/posts', { query: { page:1, tags:['a','b'] } });
 */
export async function apiRequest<TResponse, TBody = unknown>(
  url: string,
  init: ApiRequestInit<TBody> = {}
): Promise<TResponse> {
  const { body, headers, method = 'GET', ...rest } = init;

  const finalHeaders = new Headers(headers);
  let finalBody: BodyInit | undefined = undefined;

  if (body !== undefined) {
    if (isFormData(body)) {
      finalBody = body as unknown as BodyInit;
      // FormData는 Content-Type 자동 세팅(절대 수동 지정 금지)
    } else {
      finalHeaders.set('Content-Type', 'application/json');
      finalBody = JSON.stringify(body);
    }
  }

  const res = await fetch(url, { method, headers: finalHeaders, body: finalBody, ...rest });
  if (!res.ok) {
    const text = await res.text().catch(() => '');
    throw new Error(`API ${method} ${url} ${res.status}: ${text}`);
  }

  const ct = res.headers.get('content-type') || '';
  if (ct.includes('application/json')) return (await res.json()) as TResponse;
  // 필요 시 다른 타입(add): Blob, text 등
  return (await res.text()) as unknown as TResponse;
}

export async function apiGet<TResponse>(url: string, opts?: { query?: Query; init?: ApiRequestInit }) {
  const qs = toQueryString(opts?.query);
  return apiRequest<TResponse>(`${url}${qs}`, { method: 'GET', ...(opts?.init || {}) });
}


잘못된 사용 예 정리 (LSP 경고 원인)

src/lib/auth.ts (pages.signIn 지정 필수)

import type { NextAuthOptions } from "next-auth";
// ... providers 생략
export const authOptions: NextAuthOptions = {
  // providers: [...]
  pages: { signIn: "/login" },
  callbacks: {
    async jwt({ token, user }) {
      if (user) token.role = (user as any).role;
      return token;
    },
    async session({ session, token }) {
      (session.user as any).role = token.role;
      return session;
    },
  },
};


결과: 로그인 안 한 사용자 → /login?callbackUrl=... 로 302 이동. 404가 절대 안 나옴.

(C) 실제 비밀 페이지: “미로그인 → 로그인”, “로그인+비관리자 → 404”

SPA 깨지지 않도록 서버 컴포넌트에서 처리

src/app/_superadmin/page.tsx

import { getServerSession } from "next-auth";
import { redirect, notFound } from "next/navigation";
import { authOptions } from "@/lib/auth";

export const dynamic = "force-dynamic";
export const runtime = "nodejs"; // Prisma 사용하는 페이지는 Edge 금지

export default async function Page(){
  const session = await getServerSession(authOptions);
  const superPath = process.env.SUPERADMIN_PATH || "/_superadmin";

  if (!session) redirect(`/login?callbackUrl=${encodeURIComponent(superPath)}`);

  if ((session.user as any)?.role !== "ADMIN") {
    // 존재 숨기기
    notFound();
  }

  // ... 관리자 UI 렌더
  return <div className="p-6">Super Admin</div>;
}


이렇게 하면 비로그인 사용자의 모든 별칭 경로 접근은 middleware 단계에서 로그인 화면으로 보냄 → 404 발생 불가.
로그인했지만 ADMIN이 아니면 notFound() 로 존재 자체를 숨김.

4) “Fail. Route aliases… window.location.href…” 경고 원인과 차단 요약

원인: 클라이언트에서 window.location.href = '/_superadmin' 같은 직접 이동 사용 → SPA 깨짐 / 타이밍 이슈로 404
대응: 절대 금지. 서버 컴포넌트 redirect() 또는 클라이언트 router.replace('/_superadmin')만 사용.

원인: 미로그인 사용자가 별칭 경로 접근 시 페이지 코드가 먼저 실행되며 notFound()가 호출 → 404
대응: next-auth/middleware 로 선제 가드. 미로그인은 무조건 /login 으로 302. 페이지까지 가지도 않음.

보안: 최종적으로 백엔드(서버) 가드가 진짜 보안.
별칭·리다이렉트는 UX + 우회 방지. 우리는 미들웨어 + 서버 페이지 가드 이중 적용으로 완전 차단.
